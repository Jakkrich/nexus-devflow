# Markdown Metadata Contract

This contract defines the default metadata and heading-tag standard for every Markdown file generated by Nexus-DevFlow agents, workflows, and templates.

The goal is to keep Markdown useful in three places at the same time:

- Human reading in GitHub, editors, and reviews.
- Obsidian vault navigation, graph search, and Dataview-style querying.
- Agent and script indexing without forcing every report type into the same shape.

## Rule

Every generated `.md` file should use a hybrid metadata model:

1. A shared YAML frontmatter core at the top of the file.
2. Optional type-specific frontmatter fields based on `doc_type` and `category`.
3. Obsidian-compatible heading tags at the end of headings for section-level meaning.

Do not make every Markdown document identical. Make every document share the same core metadata, then add only the fields and heading tags that fit that document type.

## Required Frontmatter Core

Every generated Markdown file should start with this frontmatter block:

```yaml
---
id: "md-YYYY-MM-DD-slug"
title: "Human-readable title"
doc_type: "report"
category: "general"
status: "draft"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
owner: "agent-or-human"
source_workflow: null
related_task: null
related_files: []
tags:
  - nexus-devflow
aliases: []
summary: "One or two sentence summary."
metadata_version: 1
---
```

Required field meanings:

| Field | Type | Meaning |
| :--- | :--- | :--- |
| `id` | string | Stable unique identifier for this document. Prefer `md-YYYY-MM-DD-slug` or a workflow-specific id. |
| `title` | string | Human-readable document title. Should match or closely mirror the H1. |
| `doc_type` | string | Broad document kind, such as `spec`, `plan`, `prd`, `report`, `research`, `roadmap`, `wiki`, `decision`, `lesson`, `guide`, `changelog`, or `template`. |
| `category` | string | Workflow or knowledge category, such as `qa`, `debug`, `planning`, `review`, `release`, or `wiki`. |
| `status` | string | Lifecycle state. Use `draft`, `active`, `reviewed`, `accepted`, `superseded`, or `archived`. |
| `created` | date string | Creation date in `YYYY-MM-DD` format. |
| `updated` | date string | Last meaningful update date in `YYYY-MM-DD` format. |
| `owner` | string | Responsible agent, role, or human. |
| `source_workflow` | string or null | Workflow command that produced the document, such as `/50-Verify`, or `null` if not workflow-generated. |
| `related_task` | string or null | Task id, issue id, PR id, or `null`. |
| `related_files` | string array | Repository paths that this document describes or depends on. |
| `tags` | string array | Obsidian tags without `#`. Use hierarchical tags when useful. |
| `aliases` | string array | Alternative Obsidian note names. |
| `summary` | string | Short summary for search, indexes, and generated tables. |
| `metadata_version` | number | Contract version. Start with `1`. |

## Optional Frontmatter By Document Type

Add type-specific fields only when they fit the document. Keep absent fields absent unless a template or validator requires a blank value.

### QA Or Verification Report

```yaml
qa_result: "pending"
verification_commands:
  - "npm run validate"
risk_level: "medium"
manual_verification_required: false
```

Allowed `qa_result` values: `pass`, `fail`, `pending`, `blocked`.

Allowed `risk_level` values: `critical`, `high`, `medium`, `low`, `none`.

### Task Specification

```yaml
requirement_count: 0
acceptance_criteria_count: 0
```

Use `doc_type: "spec"` and `category: "planning"` for `.workspaces/specs/{ID}-*/20-spec.md`.

### Implementation Plan

```yaml
source_spec: ".workspaces/specs/{task-id}/20-spec.md"
phase_count: 0
risk_level: "medium"
```

Use `doc_type: "plan"` and `category: "planning"` for `.workspaces/specs/{ID}-*/30-plan.md`.

### Research Report

```yaml
research_topic: "Topic being researched"
scope: "Local codebase, external sources, or both"
confidence: "medium"
sources_count: 0
```

Allowed `confidence` values: `high`, `medium`, `low`.

### Decision Or ADR

```yaml
decision_status: "proposed"
decision_date: "YYYY-MM-DD"
supersedes: null
superseded_by: null
```

Allowed `decision_status` values: `proposed`, `accepted`, `rejected`, `superseded`.

### Debug Or RCA Report

```yaml
incident_id: null
severity: "medium"
root_cause: null
fixed_by: null
```

Allowed `severity` values: `critical`, `high`, `medium`, `low`.

### PR Review Or Specialist Report

```yaml
review_result: "pending"
review_scope: "Files, behavior, tests, and risks reviewed"
finding_count: 0
```

Allowed `review_result` values: `approve`, `request_changes`, `comment`, `pending`.

## Heading Tag Contract

Heading tags describe the meaning of a section or finding. They are Obsidian inline tags and should appear at the end of the heading text.

Use frontmatter for document-level metadata. Use heading tags for section-level metadata.

```markdown
# QA Verification Report #doc/report #report/qa

## Summary #section/summary

## Context #section/context

## Findings #section/findings

### Missing validation command #finding/gap #priority/high

## Evidence #section/evidence

## Decision #section/decision

## Follow-ups #section/followup

## Sources #section/sources
```

Heading tag rules:

- Use `#` tags only in Markdown body headings, not in frontmatter `tags`.
- Put heading tags at the end of the heading.
- Prefer lowercase hierarchical tags: `#section/summary`, `#priority/high`.
- Use ASCII tag names for portability. Avoid spaces in tags.
- Do not overload tags with long prose. Put details in the section body.

## Standard Report Template

Use this template for new generated reports unless a more specific template exists:

```markdown
---
id: "md-YYYY-MM-DD-report-slug"
title: "Report Title"
doc_type: "report"
category: "general"
status: "draft"
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
owner: "codex"
source_workflow: null
related_task: null
related_files: []
tags:
  - nexus-devflow
  - report
aliases: []
summary: "Short report summary."
metadata_version: 1
---

# Report Title #doc/report

## Summary #section/summary

Write the outcome and why it matters.

## Context #section/context

Describe scope, inputs, workflow, task id, and constraints.

## Details #section/details

Record the main analysis or report body.

## Findings #section/findings

List findings only when the report has findings. Use `#finding/*` and `#priority/*` on finding headings.

## Evidence #section/evidence

Link commands, files, commits, artifacts, screenshots, or external sources used as evidence.

## Decision #section/decision

State the current decision, recommendation, or result.

## Follow-ups #section/followup

- [ ] Add any follow-up work that should become a task, issue, or verification step.

## Sources #section/sources

- Link related artifacts, source files, commits, or references.
```

## Obsidian Compatibility Notes

- Obsidian frontmatter tags must not include `#`.
- Obsidian body tags must include `#`.
- Wikilinks are allowed for notes inside a vault: `[[Architecture Decision]]`.
- Repository paths should remain normal Markdown links or inline code paths unless the file is known to live inside the same Obsidian vault.
- Dates should use `YYYY-MM-DD` unless a workflow specifically needs date-time precision.
- YAML strings that contain `:` or workflow commands should be quoted.

## Agent Checklist

Before creating or updating a generated Markdown file:

- Add the required frontmatter core.
- Choose the closest `doc_type` and `category`.
- Add only relevant type-specific fields.
- Put document-level tags in frontmatter without `#`.
- Put section-level tags at the end of headings with `#`.
- Include `Summary`, `Context`, and `Sources` sections unless the target template has a stronger reason not to.
- Keep `updated` current when making meaningful content changes.
- Link source artifacts, commands, or files in `Sources`.
